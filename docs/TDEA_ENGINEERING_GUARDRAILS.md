# TDEA 工程防錯備忘錄

本文件是 TDEA 專案的固定工程守則。後續任何修改都必須先確認沒有破壞這裡列出的核心流程。

## 最高原則

TDEA 不是商城，也不是公開拉新系統。它是封閉協會的會員服務系統，核心價值只有三件事：

1. LINE 關鍵字要穩定回應。
2. 母站資料要能正確讀取與對應。
3. 活動報名、簽到、核銷、點數折抵要可追蹤。

任何 UI 調整、活動功能、點數功能、CRM 功能，都不能破壞上述三件事。

## 絕對不能再犯的錯

### 1. 不可讓不同管理員看到不同活動資料

活動、每月活動、報名、核銷資料是協會共享資料，不是個人工作區。

禁止：
- 用目前登入者 UID 或 email 當活動資料命名空間。
- 讓不同管理員各自建立各自的活動資料。
- 用 localStorage 當活動資料主來源。

必須：
- 活動資料以 Worker / D1 / 後端 API 為主。
- 管理員登入只做權限驗證，不影響資料分區。
- 修改活動資料後，要用另一個管理員帳號確認看到相同內容。

### 2. 不可破壞 LINE 關鍵字回應

關鍵字是 TDEA 的入口，不是附屬功能。

固定規則：
- 子站關鍵字一律以 `TDEA` 開頭。
- 母站關鍵字與子站關鍵字不可互相阻擋。
- 聊天室有文字訊息進來時，必須巡查關鍵字規則。
- 關鍵字比對必須早於草稿狀態、問答流程、暫存 session 判斷。
- 未命中子站關鍵字時，不可吞掉訊息，必須讓母站/舊流程有機會處理。

禁止：
- 因為使用者正在某個問答狀態，就直接忽略 `TDEA...` 關鍵字。
- 改活動、CRM、點數時順手改關鍵字路由。
- 用單一 if/else 把母站與子站關鍵字互斥掉。
- 未驗證就部署 webhook 關鍵字相關修改。

關鍵字修改前必查：
- `TDEA每月活動`
- `TDEA廠商列表`
- `TDEA廣告贈點`
- `會員專區`（母站）
- `會員查詢`（母站）
- `會員報到`（母站）

驗證方式：
- 每個關鍵字至少送一次真實 LINE 訊息。
- 不能只看本機或只看後台畫面。
- 回應時間若超過 2 秒，要檢查是否有不必要的同步等待。

### 3. 母站註冊資料不可再改成空資料來源

母站註冊資料頁的可用資料來源是：

`/api/aiwe-members-public`

這個來源曾確認可讀到母站名單資料。前端母站註冊資料頁必須優先使用這個來源。

禁止：
- 未確認有資料前，把母站註冊資料頁改讀新的空 API。
- 用 `/api/mother-register/records` 取代 `/api/aiwe-members-public`，除非已確認該 API 有完整資料。
- 因為欄位名稱不同就顯示空白。

必須支援的母站欄位：
- 姓名：`rosterName`, `displayName`, `name`, `nickname`
- 會員編號：`memberNo`, `rosterMemberNo`, `member_no`, `aiweMemberNo`
- LINE UID：`lineUserId`, `uid`, `line_user_id`
- 電話：`phone`, `mobile`, `tel`
- Email：`email`, `mail`, `userEmail`
- 公司/單位：`companyName`, `company`, `unit`
- 類型/來源：`rosterType`, `source`, `qualification`

顯示規則：
- 電話和 Email 必須分欄。
- 姓名欄只顯示姓名，不可把長 UID 塞在姓名下面。
- LINE UID 使用短顯示，避免表格撐爆。
- 不要把 HTTP 狀態、shop/client 這類工程資訊放在主列表。

驗證方式：
- 後端直接查 `/api/aiwe-members-public`，確認筆數大於 0。
- GitHub Pages live 端確認 `app.js` 仍包含 `/api/aiwe-members-public`。
- 後台畫面確認母站註冊資料筆數不是 0。
- 搜尋姓名、電話、UID 都要能找到資料。

## 修改前檢查表

只要要改 TDEA，先確認以下問題：

1. 這次修改是否碰到 `src/monthly-entry.ts` 或 webhook 入口？
2. 是否碰到關鍵字比對、replyToken、forward webhook、母站/子站路由？
3. 是否碰到母站註冊資料 API 或欄位對應？
4. 是否碰到活動資料來源、每月活動組裝、報名表連結？
5. 是否碰到點數、報到、核銷？

如果答案有任何一個是「是」，就必須做最小修改，且完成後跑下方驗證。

## 修改後必驗清單

### 關鍵字

- `TDEA每月活動` 有回應。
- `TDEA廠商列表` 有回應。
- `TDEA廣告贈點` 有回應。
- `會員專區` 母站有回應。
- `會員查詢` 母站有回應。
- `會員報到` 母站有回應。

### 母站註冊資料

- 母站註冊資料頁打開後不是 0 筆。
- 電話與 Email 分欄顯示。
- 姓名欄不出現長 UID。
- 搜尋可用。
- CSV 下載仍可用。

### 活動與每月活動

- 不同管理員看到同一份活動。
- 每月活動能載入上架活動。
- 每月活動預覽使用活動主圖/圖集，不混入錯誤圖片。
- LINE 關鍵字可觸發每月活動 Flex。

### 報名與核銷

- 點我報名可開啟 LIFF 報名頁。
- 會員快速報名不對錯人。
- 非會員或偵測不到 UID 時可走完整填表。
- 掃碼核銷不空白。
- 報名資料可下載 Excel。

### 點數

- 點數以母站為主。
- 子站扣點/贈點必須回寫母站。
- 折抵名單可下載明細。
- 同日打卡贈點不可母站、子站重複給點。

## 部署與驗證規則

TDEA 有兩個常見 live surface：

1. GitHub Pages：`https://fangwl591021.github.io/tdea-worker/`
2. Cloudflare Worker：`https://tdeawork.fangwl591021.workers.dev`

修改前端後，要確認 GitHub Pages 實際載到新版：

- `index.html` 的版本號要更新。
- `public/admin-login.js` 的 app 版本號要更新。
- `public/app.js` 內容要能在 GitHub Pages live URL 查到。

不能只確認 Worker，也不能只確認本機檔案。

## 禁止事項

- 禁止在修 UI 時順手改關鍵字路由。
- 禁止在修活動時順手改母站註冊資料來源。
- 禁止在未驗證 live GitHub Pages 的情況下回報已完成。
- 禁止把測試用 localStorage 狀態當成正式資料。
- 禁止用登入者身份切分協會共享資料。
- 禁止把空 API 接到正式頁面。

## 建議拆分方向

`monthly-entry.ts` 不應再繼續無限制膨脹。後續應逐步拆出：

- `keyword-router`：只處理母站/子站關鍵字判斷與轉發。
- `mother-roster`：只處理母站註冊資料讀取、欄位轉換、CRM 對應。
- `activity-service`：只處理活動、每月活動、報名資料。
- `points-service`：只處理母站點數、折抵、打卡贈點。
- `checkin-service`：只處理活動簽到與核銷。

拆分前必須先保留既有行為，不可一邊拆分一邊改規則。

## 固定結論

關鍵字和母站註冊資料是 TDEA 的底層入口。除非任務明確要求，後續任何功能修改都不得碰這兩塊。

若不得不碰，必須先列出影響範圍，再做最小修改，最後逐項驗證本文件的檢查表。
